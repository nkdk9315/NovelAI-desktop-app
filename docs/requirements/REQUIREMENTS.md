# NovelAI Desktop App - 要件定義書

## 1. 概要

NovelAI画像生成APIを利用したデスクトップアプリケーション。プロジェクト単位で画像を生成・管理し、プロンプトの組み合わせを再利用可能なグループとして管理できる。

### 対象ユーザー

NovelAIのAPIを利用して画像生成を行うクリエイター。

### プロジェクトタイプ

| タイプ | 説明 | 実装状況 |
|--------|------|----------|
| 単純生成 | プロンプト入力 → 画像生成 → 保存選択 | **初回スコープ** |
| 漫画 | 漫画制作用。使用しない画像はプロジェクト保存時に削除 | 将来実装 |
| CG集 | CG集制作用 | 将来実装 |

---

## 2. 技術スタック

| レイヤー | 技術 |
|----------|------|
| Desktop Shell | Tauri 2 |
| Backend | Rust (Tauri commands) |
| Frontend | React + TypeScript |
| UIライブラリ | Tailwind CSS + shadcn/ui |
| State管理 | Zustand |
| DB | SQLite (rusqlite, bundled) |
| API Client | `novelai-api` crate (path dependency: `novelai_api_client/rust-api`) |
| 対応OS | macOS / Windows / Linux |

## 3. 機能要件

### F1: プロジェクト管理

プロジェクトは画像生成作業の単位。ユーザーが指定したディレクトリに画像を保存する。

| 要件ID | 説明 |
|--------|------|
| F1.1 | プロジェクトの作成（名前、タイプ、保存先ディレクトリ指定） |
| F1.2 | プロジェクト一覧表示 |
| F1.3 | プロジェクトを開く |
| F1.4 | プロジェクトの削除 |
| F1.5 | プロジェクトを開いた時、未保存画像（`is_saved = false`）のファイルとDBレコードを削除 |
| F1.6 | 単純生成プロジェクトでは、全画像保存・個別保存・複数選択一括保存が可能 |
| F1.7 | 右パネル履歴一覧でサムネイルにチェックボックスを表示し、複数画像を選択して一括保存できる |

#### 画像保存方針

- **単純生成**: 生成された画像はデフォルトで未保存状態。ユーザーが明示的に保存を選択するか、全保存を選べる。
- **漫画（将来）**: プロジェクトに使用されない画像はプロジェクト保存時に削除。
- **共通**: プロジェクト再オープン時、未保存画像は容量削減のため削除される。

### F2: 画像生成

NovelAI APIを通じた画像生成の中核機能。

| 要件ID | 説明 |
|--------|------|
| F2.1 | txt2img生成（メインプロンプト + キャラクタープロンプト → API送信 → 画像受信） |
| F2.2 | 生成された画像を画面中央に表示 |
| F2.3 | 生成履歴を画面右側に縦スクロールで表示。クリックで中央に表示。ホバーでチェックボックスを表示して複数選択可能 |
| F2.4 | 生成パラメータの設定（下記「生成パラメータ」参照） |
| F2.5 | 生成時のプロンプト・パラメータのスナップショットをJSON形式で保存（再現性確保） |
| F2.6 | 生成中はローディング状態を表示 |
| F2.7 | 画像はプロジェクトディレクトリ内の `images/` サブディレクトリに保存 |
| F2.8 | **トークン数バリデーション**: メインプロンプト + 全キャラクタープロンプトのポジティブ合計 / ネガティブ合計がそれぞれ 512 トークン以下であること。UI 上にリアルタイムカウンターを表示し、超過時は生成ボタンを無効化してメッセージを表示する。T5 トークナイザー（novelai-api crate）でカウント |

#### 生成パラメータ

| パラメータ | 型 | デフォルト | 範囲 |
|------------|-----|-----------|------|
| Model | enum | `nai-diffusion-4-5-full` | 4モデル（下記参照） |
| Width | u32 | 832 | 64-2048, width*height ≤ 3,145,728 |
| Height | u32 | 1216 | 64-2048, width*height ≤ 3,145,728 |
| Steps | u32 | 23 | 1-50 |
| Scale (CFG) | f64 | 5.0 | 0.0-10.0 |
| CFG Rescale | f64 | 0.0 | - |
| Sampler | enum | `k_euler_ancestral` | 6種（下記参照） |
| Noise Schedule | enum | `karras` | 4種（下記参照） |
| Seed | u64 | ランダム | 0 - 4,294,967,295 |
| 生成枚数 | u32 | 1 | - |

#### 利用可能モデル

| モデルID | 表示名 |
|----------|--------|
| `nai-diffusion-4-curated-preview` | NAI Diffusion V4 Curated Preview |
| `nai-diffusion-4-full` | NAI Diffusion V4 Full |
| `nai-diffusion-4-5-curated` | NAI Diffusion V4.5 Curated |
| `nai-diffusion-4-5-full` | NAI Diffusion V4.5 Full |

#### サンプラー

`k_euler`, `k_euler_ancestral`, `k_dpmpp_2s_ancestral`, `k_dpmpp_2m_sde`, `k_dpmpp_2m`, `k_dpmpp_sde`

#### ノイズスケジュール

`native`, `karras`, `exponential`, `polyexponential`

### F3: プロンプトシステム

APIの`v4_prompt`構造に基づく二層プロンプトシステム。

| 要件ID | 説明 |
|--------|------|
| F3.1 | **メインプロンプト**: 背景、場所、カメラワーク等を記述。ポジティブ/ネガティブの2フィールド |
| F3.2 | **キャラクタープロンプト**: キャラクターごとの外見・ポーズ等。ポジティブ/ネガティブの2フィールド |
| F3.3 | キャラクターごとに画像内の位置を指定（center_x, center_y: 0.0-1.0） |
| F3.4 | メインプロンプトのネガティブにはデフォルト値を初期設定 |
| F3.5 | プロンプト入力時、システムプロンプト（F7）からの予測変換/オートコンプリート |
| F3.6 | プロンプトグループ（F4）から選択してプロンプトにタグを追加（累積的に追加可能） |

#### デフォルトネガティブプロンプト

```
nsfw, lowres, artistic error, film grain, scan artifacts, worst quality, bad quality, jpeg artifacts, very displeasing, chromatic aberration, dithering, halftone, screentone
```

#### v4_prompt構造（API送信形式）

```json
{
  "v4_prompt": {
    "caption": {
      "base_caption": "<メインプロンプト>",
      "char_captions": [
        {
          "char_caption": "<キャラクター1プロンプト>",
          "centers": [{"x": 0.5, "y": 0.5}]
        }
      ]
    },
    "use_coords": true,
    "use_order": true
  },
  "v4_negative_prompt": {
    "caption": {
      "base_caption": "<メインネガティブ>",
      "char_captions": [
        {"char_caption": "<キャラクター1ネガティブ>", "centers": [{"x": 0.5, "y": 0.5}]}
      ]
    },
    "legacy_uc": false
  }
}
```

### F4: プロンプトグループ管理

再利用可能なプロンプトタグの集合を「プロンプトグループ」として管理する。

| 要件ID | 説明 |
|--------|------|
| F4.1 | グループの作成（名前、所属ジャンル、使用タイプ、タグリスト） |
| F4.2 | グループの編集（名前、ジャンル、タグの追加/削除/並び替え） |
| F4.3 | グループの削除（システムグループは削除不可） |
| F4.4 | グループ一覧表示（ジャンルでフィルタ、使用タイプでフィルタ、名前で検索） |
| F4.5 | グループにデフォルトフラグを設定（ジャンルごとに1つ。同ジャンルの既存デフォルトは解除） |
| F4.6 | プロンプトグループ管理画面（モーダル）でCRUD操作 |

#### グループ属性

| 属性 | 説明 |
|------|------|
| name | グループ名 |
| genre_id | 所属ジャンル（F5のジャンル） |
| is_default_for_genre | このジャンルのデフォルトとして自動適用するか |
| is_system | システムグループ（編集/削除不可） |
| usage_type | `main`（メインプロンプト用）、`character`（キャラクタープロンプト用）、`both`（両方） |
| tags | タグのリスト（順序保持）。例: `["sitting", "spread legs"]`, `["fighting stance", "bent knee"]` |

#### 使用タイプの柔軟性

- `usage_type` はグループの「推奨用途」であり、制約ではない。
- メイン用に作られたグループもキャラクタープロンプトとして使用可能（UIでフィルタ/ソート用）。
- 逆にキャラクター用グループもメインプロンプトに適用可能。

### F5: ジャンル管理

プロンプトグループの分類に使うジャンル（カテゴリ）。

| 要件ID | 説明 |
|--------|------|
| F5.1 | システムジャンル: `男`, `女`, `その他` (削除不可) |
| F5.2 | ユーザーがカスタムジャンルを作成可能（例: 宇宙人、犬、ロボット等） |
| F5.3 | カスタムジャンルの削除 |
| F5.4 | ジャンル一覧表示（ソート順あり） |

### F6: キャラクター管理

画像生成時に最大6人のキャラクターを追加・管理する。

| 要件ID | 説明 |
|--------|------|
| F6.1 | キャラクター追加ボタン（男/女/その他/カスタムジャンル） |
| F6.2 | ボタン押下時、そのジャンルのデフォルトプロンプトグループ（F4.5）を自動適用 |
| F6.3 | 各キャラクターにプロンプトグループを追加選択可能 |
| F6.4 | キャラクターの削除 |
| F6.5 | キャラクター追加上限: 6人（API制約: `MAX_CHARACTERS = 6`） |
| F6.6 | 各キャラクターの画像内位置指定（center_x, center_y スライダー） |

### F7: システムプロンプト（タグデータベース）

danbooruタグのCSVデータベースをアプリにバンドルし、予測変換とグループとして利用する。

| 要件ID | 説明 |
|--------|------|
| F7.1 | CSVファイル（167,907件）をアプリリソースとしてバンドル |
| F7.2 | 起動時にRust側でメモリにロード |
| F7.3 | プロンプト入力時の予測変換/オートコンプリート（部分一致検索、デバウンス付き） |
| F7.4 | カテゴリごとにグループとして選択可能 |
| F7.5 | システムプロンプトはユーザーが編集/削除不可 |

#### CSVデータ形式

```
tag_name,category,post_count,aliases
```

例: `1girl,0,7553466,"sole_female,1girls"`

#### カテゴリマッピング

| カテゴリID | 名称 | 件数 |
|-----------|------|------|
| 0 | General（一般タグ） | 47,756 |
| 1 | Artist（アーティスト） | 60,631 |
| 2 | (その他) | 166 |
| 3 | Copyright（作品名） | 10,207 |
| 4 | Character（キャラクター名） | 43,729 |
| 5 | Meta（メタタグ） | 4,879 |
| 6 | (その他) | 2 |
| 7 | (その他) | 437 |
| 8 | (その他) | 100 |

#### 検索仕様

- Rust側でインメモリ検索（DB不要、高速）
- 部分一致（substring match）
- post_countで人気順ソート
- aliases も検索対象に含める
- フロントエンドからは Tauri コマンド経由でデバウンス付きクエリ

### F8: Vibe管理

`.naiv4vibe` ファイルを管理し、画像生成時にスタイル転写として使用する。

| 要件ID | 説明 |
|--------|------|
| F8.1 | `.naiv4vibe` ファイルのインポート（ファイルピッカーで選択、アプリデータディレクトリにコピー） |
| F8.2 | Vibe一覧表示（名前、モデル、作成日時） |
| F8.3 | Vibeの削除（ファイル + DBレコード） |
| F8.4 | 画像生成画面の左パネルでVibeのON/OFF選択 |
| F8.5 | Vibeごとの strength（0.0-1.0、デフォルト: 0.7）と information_extracted（0.0-1.0、デフォルト: 0.7）の調整 |
| F8.6 | 最大10個のVibeを同時使用可能（API制約: `MAX_VIBES = 10`） |
| F8.7 | Vibeのエンコード（画像 → `.naiv4vibe` ファイル生成。API経由、2 Anlas消費） |

#### .naiv4vibe ファイル形式

```json
{
  "encodings": {
    "v4-5full": { "<uuid>": { "encoding": "<base64>", "params": { "information_extracted": 0.7 } } },
    "v4-5curated": { ... },
    "v4full": { ... },
    "v4curated": { ... }
  },
  "importInfo": { "information_extracted": 0.7 },
  "source_image_hash": "<sha256>",
  "created_at": "2024-01-01T12:00:00Z"
}
```

### F10: Anlas管理

サイト内通貨「Anlas」の残高表示とコスト予測。

| 要件ID | 説明 |
|--------|------|
| F10.1 | Anlas残高の取得・表示（API: `GET /user/subscription`） |
| F10.2 | 残高の内訳表示: 固定（fixed） + 購入分（purchased） = 合計 |
| F10.3 | サブスクリプションプラン表示（tier: 0=Free, 1=Tablet, 2=Scroll, 3=Opus） |
| F10.4 | 生成前のコスト予測表示（純粋関数による計算、API呼び出し不要） |
| F10.5 | パラメータ変更時にリアルタイムでコスト更新（デバウンス付き） |
| F10.6 | Anlas不足時の警告表示 |

#### Opusプラン無料生成条件

- キャラクターリファレンス未使用
- Vibe未使用
- width * height ≤ 1,048,576 (1024x1024)
- steps ≤ 28
- tier ≥ 3 (Opus)

#### コスト計算式（V4モデル）

```
base_cost = (2.951823174884865e-6 × pixels) + (5.753298233447344e-7 × pixels × steps)
vibe_batch_cost = max(0, (vibe_count - 4) × 2)  // 4個まで無料
char_ref_cost = 5 × char_ref_count × n_samples
vibe_encode_cost = 2 // エンコードごと
```

## 6. 非機能要件

### セキュリティ

- APIキーはSecretString相当でメモリ上に保持
- DBに保存する場合は暗号化を検討
- パストラバーサル防御はAPIクライアント側で実装済み

### パフォーマンス

- システムプロンプト（167,907件）はRust側インメモリ検索で高速応答
- 画像表示: 現在画像はbase64 over IPC、履歴サムネイルは将来的にTauriアセットプロトコルに最適化可
- コスト計算: 純粋関数でUI応答性を維持

### データ永続化

- SQLite DBはアプリデータディレクトリ（`$APPDATA/novelai-desktop/app.db`）に保存
- Vibeファイルは`$APPDATA/novelai-desktop/vibes/`に保存（プロジェクト横断で共有）
- プロジェクト画像はユーザー指定ディレクトリ内の`images/`に保存
- DB内の画像パスはプロジェクトディレクトリからの相対パス（ポータビリティ確保）

### 将来拡張性

- プロジェクトタイプの追加（漫画、CG集）
- img2img / inpaint 機能
- Augment ツール（colorize, declutter, emotion, sketch, lineart, bg-removal）
- Upscale（2x / 4x）
- キャラクターリファレンス（参照画像からの生成）
