# 品質スコアリング基準

## ゴールデンルール

以下は本プロジェクトにおける絶対遵守事項。違反は即修正対象。

1. **レイヤー境界を越えない** — Commands が Repositories を直接呼ばない
2. **SQL は必ずパラメータバインド** — 文字列連結による SQL 構築禁止
3. **AppError の適切なバリアント使用** — catch-all にしない（NotFound / Validation / Database を正しく使い分け）
4. **1 ファイル 300 行以内** — 超過時はドメイン別に分割
5. **UUID・タイムスタンプは Rust 側で生成** — Frontend で生成しない
6. **API キーは SecretString で保持** — ログ・Debug 出力に平文で漏らさない
7. **フロントエンドの状態は Store に集約** — コンポーネントローカル state は UI 状態のみ

## スコアリング基準（5 カテゴリ × 5 点 = 25 点満点）

### 1. 正確性 (Correctness) — 5 点

| 点数 | 基準 |
|------|------|
| 5 | 全テストケース通過、エッジケース網羅、型安全 |
| 4 | 主要パス正常、軽微なエッジケース漏れ |
| 3 | 基本動作するが既知の制約あり |
| 2 | 一部パスで不正動作 |
| 1 | 主要機能が壊れている |

### 2. セキュリティ (Security) — 5 点

| 点数 | 基準 |
|------|------|
| 5 | SecretString 使用、パラメータバインド、CSP 適切、パストラバーサル防御 |
| 4 | 主要セキュリティ対策あり、軽微な改善余地 |
| 3 | 基本対策あるが一部不足 |
| 2 | 明確なセキュリティ問題あり |
| 1 | 認証情報漏洩やインジェクション脆弱性 |

### 3. パフォーマンス (Performance) — 5 点

| 点数 | 基準 |
|------|------|
| 5 | 不要な clone/allocation 最小、適切なインデックス使用、Mutex 保持期間最短 |
| 4 | 主要パスは最適化済み |
| 3 | 動作するが最適化余地あり |
| 2 | 明確なボトルネックあり（N+1 クエリ等） |
| 1 | ブロッキング操作やメモリリーク |

### 4. 保守性 (Maintainability) — 5 点

| 点数 | 基準 |
|------|------|
| 5 | 300 行以内、明確な責務分離、命名規則準拠 |
| 4 | 構造良好、軽微な改善余地 |
| 3 | 理解可能だが冗長な部分あり |
| 2 | 複雑な条件分岐、責務混在 |
| 1 | スパゲッティコード |

### 5. テストカバレッジ (Test Coverage) — 5 点

| 点数 | 基準 |
|------|------|
| 5 | テスト対象レイヤーの全パス網羅、エッジケース含む |
| 4 | 主要パスカバー |
| 3 | 正常系のみ |
| 2 | 一部テストあるが不十分 |
| 1 | テストなし（テスト対象レイヤーの場合） |

## 合格基準

| レベル | 条件 |
|--------|------|
| **最低** | 各カテゴリ 3 点以上、合計 18 点以上 |
| **推奨** | 各カテゴリ 4 点以上、合計 22 点以上 |
| **必須** | セキュリティは 4 点以上 |

## 簡易チェックリスト（PR レビュー用）

- [ ] レイヤー境界遵守（Commands → Services → Repositories）
- [ ] SQL パラメータバインド使用
- [ ] API キーに SecretString 使用
- [ ] AppError の適切なバリアント選択
- [ ] 1 ファイル 300 行以内
- [ ] テスト対象レイヤーのテスト追加/更新
- [ ] `unwrap()` / `expect()` がプロダクションコードに無い
- [ ] `any` 型がフロントエンドコードに無い
- [ ] 命名規則準拠（Rust: snake_case, TS: camelCase/PascalCase）
- [ ] innerHTML の直接操作無し
